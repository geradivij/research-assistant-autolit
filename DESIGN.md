## Problem Statement

autolit is a local-first, agentic literature-review pipeline: drop PDFs in, get a
structured, AI-generated survey across multiple papers. The core pipeline works
(Phase 1 ingestion, Phase 2 field-by-field extraction, Phase 3 selection/comparison/writing)
but there are two blockers to sharing it:

1. **No tests.** The pipeline's correctness is only verified by running it end-to-end
   against a live Ollama instance. A broken JSON repair, a silent fallback, or an
   off-by-one in top-k retrieval is invisible.

2. **No UI.** The only interface is Python scripts. Target audience (AI builders/hackers)
   can't see the pipeline run, can't drag-and-drop PDFs, can't share a link.

The goal is: solid tests + a beautiful, local-first web UI that makes the pipeline
demonstrable and shareable.

---

## What Makes This Cool

- **The pipeline log is the product.** Watching the selector pick papers, the comparator
  produce a table, the writer stream Markdown — that's what impresses AI builders.
  Nobody else shows this. Elicit, ResearchRabbit, and Connected Papers all hide their
  internals. autolit makes the agents visible.

- **Fully local.** No API keys, no cloud, no data leaving the machine. Ollama + FAISS +
  sentence-transformers. For researchers handling unpublished papers, that matters.

- **Field-by-field extraction.** Not "one giant JSON prompt." Each field (task, approach,
  datasets, metrics, key_results, limitations, notes) gets its own RAG query and its own
  tight LLM call. This is architecturally better than what ResearchPilot (the closest
  published system, arxiv March 2026) does. Worth calling out in the README.

---

## Constraints

- Python-only toolchain. No Node.js build step — `git clone && pip install` should be
  the full setup path for the UI.
- Runs against a local Ollama instance (llama3 or equivalent).
- Data lives in `data/pdfs/` and `data/indexes/`. The UI doesn't change this structure.
- Tests must be runnable in CI without Ollama (mocked unit tier) and optionally with
  Ollama (integration tier).

---

## Premises

1. The Phase 3 pipeline (selection + comparator + writer) is the core value. Unit tests
   with mocked LLM calls are the first priority — fast, CI-safe. Integration tests
   (`@pytest.mark.integration`) that require Ollama running are the second tier.
   *(Agreed)*

2. The frontend must ship with no Node.js toolchain. `pip install` + `python -m autolit`
   opens the browser. *(Agreed)*

3. "Very aesthetic" for this audience = dark mode, streaming pipeline output (the log
   pane is the star), clean typography in the survey view, and readable Markdown
   rendering. Not a multi-page SPA. *(Agreed)*

4. The hard testing targets are `contracts.py` (JSON parsing/repair), the selection
   agent fallback ranker (token overlap), and the extraction field loop. FAISS operations
   and chunking are library wrappers — low test priority. *(Agreed)*

## Approaches Considered

### Approach A: FastAPI + HTMX/Alpine.js + Tailwind CDN
Serve Jinja2 templates, stream pipeline progress via SSE, use Alpine.js for
client-side state. No Node.js required. Full control over styling. Medium CSS discipline
needed to look truly polished.

### Approach B: FastAPI + React via esm.sh CDN (Chosen)
FastAPI REST API + a single `index.html` that imports React, ReactDOM, and a Markdown
renderer via esm.sh CDN (no npm, no Webpack). React components for PaperDropzone,
PipelineStatus, SurveyViewer, PaperLibrary. SSE for streaming.

### Approach C: Gradio v5 + Custom CSS
Python-only, has streaming via `.queue()`, deployable in one file. But "Gradio look"
is immediately recognizable to the target audience — signals "demo," not "product."

## Recommended Approach

**Approach B — FastAPI + React via CDN.**

The FastAPI REST API is a clean separation: the pipeline logic in `src/autolit/` stays
unchanged. The frontend is a single `src/autolit/web/index.html` served by FastAPI's
StaticFiles. React via esm.sh CDN means no build step — just a `<script type="module">`
tag.

The four API routes with contracts:
```
GET  /api/papers
  Response: [{paper_id: str, status: "indexed"|"processing"|"error", has_summary: bool}]

POST /api/ingest
  Body: multipart/form-data, field "file" = PDF bytes
  Response: {paper_id: str, status: "processing"}
  Behavior: runs Phase 1 (chunk+index) + Phase 2 (field extraction) synchronously
            in a thread pool (asyncio.run_in_executor). Marks paper "indexed" on completion.
  Note: on first ingest, sentence-transformers downloads ~400MB embedding model (one-time,
        cached). The API logs: "Downloading embeddings model (one-time, ~2 min)..."
  Limit: PDFs > 100MB rejected with 413 Payload Too Large.

GET  /api/summaries/{paper_id}
  Response: {status: "indexed"|"processing"|"not_found", summary: PaperSummary | null}
  Summaries stored at outputs/summaries/{paper_id}.json (existing pipeline output).

POST /api/survey/stream
  Body: {topic: str, paper_ids: [str] | null, model: str = "llama3"}
        (null paper_ids = use all indexed papers)
  Response: text/event-stream (SSE)
  Event format: data: {"type": "log"|"result"|"error",
                        "agent": "selector"|"comparator"|"writer"|"extract",
                        "message": str,
                        "timestamp": ISO8601}
  Final event: {"type": "result", "markdown": "<full survey text>"}
  SSE strategy: pipeline runs in asyncio background task. SSE endpoint polls a
                per-request asyncio.Queue (producer=pipeline, consumer=SSE generator).
                Client reconnect: SSE events include id= fields; client uses
                Last-Event-ID header to resume. Server buffers last 100 events in memory.
```

**Error handling:** Pipeline errors (LLM timeout, JSON repair failure after 2 attempts) emit
`{type: "error", message: "..."}` events. The SurveyViewer shows a retry button on error.
The PipelineStatus bar shows the failed stage in red.

**Paper metadata:** Persisted in `data/papers_manifest.json`, keyed by paper_id:
```json
{
  "memorization": {
    "status": "indexed",
    "ingested_at": "2026-04-12T00:53:00Z",
    "error": null
  }
}
```
GET /api/papers reads this file. POST /api/ingest writes to it.
Response is immediate (`{paper_id, status: "processing"}`); client polls GET /api/papers
until status flips to "indexed" or "error". Duplicate ingests overwrite summary + re-index.
Embedding download errors return `{status: "error", error: "Embedding model download failed."}`.

**Streaming event architecture — key refactor:** The current pipeline is synchronous.
To stream agent progress, Phase 3 agents need a progress callback:
```python
def progress_callback(agent: str, message: str) -> None: ...
```
`run_survey_pipeline` accepts this callback and calls it at each major step:
- Before/after `select_papers_for_topic`
- Before/after `compare_papers`
- As `write_survey_markdown` produces output
The SSE endpoint passes an `asyncio.Queue.put_nowait` wrapper as the callback.
This is the most substantial refactor — recommended as a spike before full UI build.

The five React components:
```
<PaperDropzone>     drag-and-drop PDF upload, progress indicator
<PaperLibrary>      list of papers, colored dots (indexed/processing/error)
<PipelineStatus>    5-stage progress bar (Ingest → Extract → Select → Compare → Write)
<SurveyViewer>      rendered Markdown, dark mode, comparison table, export-to-file button
<PipelineLog>       live SSE stream, color-coded by agent:
                      selector:   amber   #f59e0b
                      comparator: violet  #a78bfa
                      writer:     sky     #0ea5e9
                      extract:    emerald #10b981
```

Initial page load (~2–3 sec) is limited by React CDN bundle. HTML skeleton renders
immediately (sidebar, empty pipeline bar) so the page doesn't feel blank during load.

**Visual design** — dark, minimal, indigo accent (`#6366f1`).
Layout: 260px left sidebar / main content with pipeline bar at top / survey + log split.

## Test Architecture

### Unit tests (no Ollama required, ~10 seconds)
Target: `tests/unit/`

| Test file | What it covers |
|---|---|
| `test_contracts.py` | `parse_or_repair_json` with valid JSON, malformed JSON, code-fenced JSON; `_extract_between`; `_strip_code_fences` |
| `test_selection_fallback.py` | `_fallback_rank_by_overlap` — correct ordering, empty topic, all-same-score tie |
| `test_extraction_fields.py` | `_parse_list_field` parsing; `_ask_field_llm` with mocked LLM; field loop iteration |
| `test_chunking.py` | Overlap logic, edge cases (empty text, single page) |

**Mock strategy:** monkeypatch `src.autolit.llm.client.chat` to return fixture strings.
Use `pytest-mock`. No Ollama, no FAISS reads required for these tests.

### Integration tests (require Ollama + llama3, ~3-5 min)
Target: `tests/integration/`
Marked with `@pytest.mark.integration` — skipped by default.

| Test | What it verifies |
|---|---|
| `test_ingest_single_pdf.py` | Ingest `data/pdfs/memorization.pdf` → FAISS index built, chunks.json written |
| `test_extraction_real.py` | `summarize_paper_structured("memorization")` → valid PaperSummary, all fields non-empty |
| `test_survey_pipeline.py` | `run_survey_pipeline("memorization")` → Markdown file written, contains section headers |

Run with: `pytest -m integration`

### CI config
Default `pytest` (no flag) = unit tests only. Fast, always green.

## Future Enhancements (out of scope for this iteration)

- **Model selector:** `/api/models` endpoint lists available Ollama models; UI shows dropdown.
  Currently hard-coded to `"llama3"`.
- **PDF progress estimation:** Per-page ingest progress. Currently the paper dot spins
  until fully indexed.
- **Report history:** SQLite for survey search + history. Currently surveys written to
  `outputs/surveys/` as Markdown files.

---

## Success Criteria

- `pytest` (unit tier) passes in < 30 seconds with zero Ollama dependency.
- `pytest -m integration` passes end-to-end using `data/pdfs/memorization.pdf`.
- `python -m autolit` opens a browser to `localhost:8000`, no Node.js required.
- Drop a PDF, enter a topic, click Run — see the pipeline log stream in real time,
  survey appears as it writes.
- A hacker who `git clone`s and `pip install`s says "whoa" within 90 seconds.

   PipelineLog → SurveyViewer + export.

8. **Integration tests** — against `data/pdfs/memorization.pdf`.

## What I noticed about how you think

- You said you wanted "1 and 3" on the vision question — zero friction AND beautiful
  reading experience. Not one or the other. That's a product instinct: the pipeline is
  the engine, the output quality is the product. Most people building AI tools optimize
  for the pipeline and treat the output as an afterthought.

- When I asked about tests, you pushed back on the mocked-only approach immediately —
  "I want full integration tests too." You didn't accept a shortcut that would give you
  false confidence. That's how you avoid the class of bugs where "mocked tests passed
  but the real pipeline broke."

- You've already shipped a meaningful amount of this. The commit history goes from PDF
  loader to multi-agent Phase 3 in ~10 commits. That's a real implementation arc, not
  a prototype that never got finished.
